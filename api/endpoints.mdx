## POST /api/schedule

Run the scheduling engine on a set of volunteers and shifts. The engine assigns volunteers to open shift slots respecting group requirements, max hours, and time-overlap constraints, then returns a fairness score and any conflicts.

**Authentication required:** Yes — `Authorization: Bearer [user_id].[signature]`

### Request body

<ParamField body="volunteers" type="object[]" required>
  List of volunteers to schedule.

  <Expandable title="volunteer object properties">
    <ParamField body="id" type="string" required>
      Unique identifier for the volunteer. Must match the IDs used in `current_assignments` if any are provided.
    </ParamField>
    <ParamField body="name" type="string" required>
      Display name of the volunteer (e.g., `"Alice"`, `"Delegate 1"`).
    </ParamField>
    <ParamField body="group" type="string" required>
      The volunteer's group affiliation. Must match a key in a shift's `required_groups` to be eligible for that shift (e.g., `"Delegates"`, `"Adults"`, `"Security"`).
    </ParamField>
    <ParamField body="max_hours" type="number" required>
      Maximum hours this volunteer can be assigned. The engine will not assign the volunteer beyond this limit. Pass `999` to indicate no limit.
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="unassigned_shifts" type="object[]" required>
  Shifts that need to be filled by the engine.

  <Expandable title="shift object properties">
    <ParamField body="id" type="string" required>
      Unique identifier for the shift.
    </ParamField>
    <ParamField body="start" type="string" required>
      ISO 8601 datetime for the shift start (e.g., `"2026-05-01T09:00:00Z"`).
    </ParamField>
    <ParamField body="end" type="string" required>
      ISO 8601 datetime for the shift end (e.g., `"2026-05-01T17:00:00Z"`).
    </ParamField>
    <ParamField body="required_groups" type="object" required>
      A map of group name to required volunteer count. For example, `{"Delegates": 2, "Adults": 2}` means 2 volunteers from the Delegates group and 2 from Adults are needed to fill the shift.
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="current_assignments" type="object[]">
  Optional. Existing assignments to lock in before the engine runs. The engine will preserve these and fill remaining slots around them.

  <Expandable title="assignment object properties">
    <ParamField body="shift_id" type="string" required>
      ID of the shift.
    </ParamField>
    <ParamField body="volunteer_id" type="string" required>
      ID of the volunteer already assigned to this shift.
    </ParamField>
  </Expandable>
</ParamField>

### Response body

<ResponseField name="assigned_shifts" type="object">
  A map of `shift_id` to an array of `volunteer_id` strings. Shifts with no assignable volunteers will not appear here — check `unfilled_shifts` for those.

  ```json
  {
    "shift_101": ["vol_7", "vol_12"],
    "shift_102": ["vol_3"]
  }
  ```
</ResponseField>

<ResponseField name="unfilled_shifts" type="string[]">
  List of `shift_id` values for shifts that could not be fully filled. Cross-reference with `conflicts` to understand why.
</ResponseField>

<ResponseField name="fairness_score" type="number">
  A float representing how evenly the workload is distributed across all volunteers. Lower values indicate a more balanced schedule. A score of `0.12` is well-distributed; values closer to `1.0` indicate significant imbalance.
</ResponseField>

<ResponseField name="conflicts" type="object[]">
  Detailed reasons for each unfilled shift.

  <Expandable title="conflict object properties">
    <ResponseField name="shift_id" type="string">
      The ID of the unfillable shift.
    </ResponseField>
    <ResponseField name="group" type="string">
      The group that could not be filled (e.g., `"Lifeguards"`).
    </ResponseField>
    <ResponseField name="reasons" type="string[]">
      Human-readable explanations for why the slot could not be filled (e.g., `"Volunteer vol_7 already reached max_hours (40.0)"`).
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="volunteers" type="object">
  A map of `volunteer_id` to a summary of their resulting schedule.

  <Expandable title="volunteer summary properties">
    <ResponseField name="assigned_hours" type="number">
      Total hours assigned to this volunteer in the resulting schedule.
    </ResponseField>
    <ResponseField name="assigned_shifts" type="string[]">
      List of `shift_id` values this volunteer was assigned to.
    </ResponseField>
  </Expandable>
</ResponseField>

### Example

<CodeGroup>

```bash cURL
curl -X POST https://shift-scheduler-api-3nxm.vercel.app/api/schedule \
     -H "Authorization: Bearer YOUR_KEY" \
     -H "Content-Type: application/json" \
     -d '{
       "volunteers": [
         {"id": "v1", "name": "Alice", "group": "Adults", "max_hours": 12},
         {"id": "v2", "name": "Delegate 1", "group": "Delegates", "max_hours": 10}
       ],
       "unassigned_shifts": [
         {
           "id": "s1",
           "start": "2026-05-01T09:00:00Z",
           "end": "2026-05-01T17:00:00Z",
           "required_groups": {"Delegates": 2, "Adults": 2}
         }
       ],
       "current_assignments": []
     }'
```

```python Python
import requests

payload = {
    "volunteers": [
        {"id": "v1", "name": "Alice", "group": "Adults", "max_hours": 12},
        {"id": "v2", "name": "Delegate 1", "group": "Delegates", "max_hours": 10}
    ],
    "unassigned_shifts": [
        {
            "id": "s1",
            "start": "2026-05-01T09:00:00Z",
            "end": "2026-05-01T17:00:00Z",
            "required_groups": {"Delegates": 2, "Adults": 2}
        }
    ],
    "current_assignments": []
}

response = requests.post(
    "https://shift-scheduler-api-3nxm.vercel.app/api/schedule",
    json=payload,
    headers={"Authorization": "Bearer YOUR_KEY"}
)
print(response.json())
```

</CodeGroup>

## POST /api/schedule/csv

Same scheduling engine as `POST /api/schedule`, but accepts CSV files via `multipart/form-data` instead of a JSON body. Useful for bulk imports from spreadsheet tools.

**Authentication required:** Yes — `Authorization: Bearer [user_id].[signature]`

### Form fields

<ParamField body="volunteers" type="file" required>
  A CSV file with volunteer data. Use the same column structure as `sample_volunteers.csv`: `Name`, `Group`, `Max Hours`, `Email`, `Phone`. The `Email` and `Phone` columns are metadata and are not used by the scheduling engine.

  ```csv
  Name,Group,Max Hours,Email,Phone
  Delegate 1,Delegates,10,delegate1@example.com,(555) 101-0001
  Adult 66,Adults,12,adult66@example.com,(555) 201-0066
  ```
</ParamField>

<ParamField body="shifts" type="file" required>
  A CSV file with shift data. Use the same column structure as `sample_shifts.csv`: `Name`, `Start`, `End`, `Groups`.

  The `Groups` column accepts a comma-separated list of `Group:Count` pairs (e.g., `"Delegates:2, Adults:2"`).

  ```csv
  Name,Start,End,Groups
  Shift 1,12/01/2025 08:00 AM,12/01/2025 10:00 AM,"Delegates:2, Adults:2"
  Shift 2,12/01/2025 10:00 AM,12/01/2025 12:00 PM,"Delegates:2, Adults:2"
  ```
</ParamField>

### Example

```bash
curl -X POST https://shift-scheduler-api-3nxm.vercel.app/api/schedule/csv \
     -H "Authorization: Bearer YOUR_KEY" \
     -F "volunteers=@sample_volunteers.csv" \
     -F "shifts=@sample_shifts.csv"
```

The response schema is identical to `POST /api/schedule`.

## POST /api/validate

Validate a request payload without running the scheduler. The API checks the structure and types of your `volunteers`, `unassigned_shifts`, and `current_assignments` fields and returns any validation errors.

**Authentication required:** Yes — `Authorization: Bearer [user_id].[signature]`

<Tip>
  Call this endpoint first when debugging a `400 Bad Request` from `/api/schedule`. It returns specific field-level errors that identify exactly what is malformed in your payload.
</Tip>

### Request body

Same schema as `POST /api/schedule` — `volunteers`, `unassigned_shifts`, and optionally `current_assignments`.

### Example

```bash
curl -X POST https://shift-scheduler-api-3nxm.vercel.app/api/validate \
     -H "Authorization: Bearer YOUR_KEY" \
     -H "Content-Type: application/json" \
     -d '{
       "volunteers": [{"id": "v1", "name": "Alice", "group": "Adults", "max_hours": 12}],
       "unassigned_shifts": [{"id": "s1", "start": "2026-05-01T09:00:00Z", "end": "2026-05-01T17:00:00Z", "required_groups": {"Adults": 1}}]
     }'
```

## GET /api/usage

Retrieve your current API quota and usage history for your API key.

**Authentication required:** Yes — `Authorization: Bearer [user_id].[signature]`

### Response body

<ResponseField name="quota" type="number">
  Your total daily request quota.
</ResponseField>

<ResponseField name="usage_count" type="number">
  Number of requests made in the current billing period.
</ResponseField>

<ResponseField name="history" type="object[]">
  A list of recent API calls with timestamps and endpoint details.
</ResponseField>

### Example

```bash
curl https://shift-scheduler-api-3nxm.vercel.app/api/usage \
     -H "Authorization: Bearer YOUR_KEY"
```
